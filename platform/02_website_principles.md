# 网站建设原则标准

这些原则在后续网站和工具建设中不能违背。若必须违背，必须先写入决策记录并说明原因。

## 产品原则

- 平台核心必须以预测市场中现存的可交易选题为中心，世界杯赛程和球队信息是解释层，不是主线。
- 第一屏必须是实际工作台，不做营销页。
- 不展示“稳赚”“必中”“无风险”等承诺。
- 所有交易建议必须同时展示风险、取消条件和数据更新时间。
- 内容输出和下注建议必须分离。
- 用户必须能看出一条建议来自数据、模型、人工判断还是内容灵感。
- 平台默认只读，任何交易执行能力都必须显式开启。
- 长期商业目标是订阅付费，但第一版必须先验证功能，不先做付费墙。
- 第一版不做账号系统和登录。
- Polymarket 外链必须统一追加联盟参数：`?via=serene77mc-g6kj`。

## 数据原则

- 每个预测市场选题必须识别结构类型：二元、多个二元互斥、单市场多选、不互斥相关、独立二元或未知。
- 同一事件下多个互斥市场必须合并看风险敞口，不能只看单个选题。
- 所有市场数据必须显示来源。
- 所有价格必须显示抓取时间。
- 数据过期必须降级或标红。
- Polymarket 数据必须优先使用官方接口和官方 Agent Skills 资料。
- 国家名不能单独作为世界杯市场匹配条件，必须出现 World Cup/FIFA/世界杯 ticker 等强信号。
- 低流动性市场不能和高流动性市场同权展示，但不能因为低交易量直接判定为低关注或无价值。
- 新上架、未到比赛窗口、未进入传播周期的事件可以暂时低交易量；只有事件已结算、关闭、规则判定结束，才标记为结束状态。
- 当前选题数量必须按最新快照去重统计；历史快照用于状态变化，不用于当前数量直接累加。

## 风险原则

- 能给下注方向时必须标记 YES / NO / WAIT / AVOID，并说明方向来源、置信度、取消条件。
- 缺少 `neg_risk` 或互斥关系字段时必须标记未知，不能假装已确认。
- 没有取消条件，不给仓位建议。
- 没有复盘字段，不允许进入策略库。
- 单场、单日、单观点必须有仓位上限。
- 相关盘口必须合并看风险敞口。
- 连续亏损必须触发自动降仓规则。
- 所有高盈亏比机会默认小仓，不因赔率高而重仓。

## 界面原则

- 界面优先服务扫描和决策，不做花哨装饰。
- 页面应像交易/研究工作台，而不是资讯站或营销站。
- 不使用大面积英雄区。
- 不使用过度装饰的卡片堆叠。
- 不做单一色调主题。
- 颜色必须有功能含义：
  - Green: 可参与。
  - Yellow: 等待信息。
  - Red: 禁止或高风险。
  - Blue: 只观察。
  - Gray: 已结束或无效。
  - Purple: 新上架或等待交易量增长。
- 表格、榜单、状态标签必须比大段说明更优先。

## 页面原则

### 今日工作台

必须显示：

- 今日比赛。
- 今日可参与市场数量。
- 最值得关注 Top 3。
- 最被忽视 Top 3。
- 最大盈亏比 Top 3。
- 稳健机会 Top 3。
- 高盈亏比观察 Top 3。
- 今日风险额度。
- 已用仓位。
- 等首发事件。
- 禁止下注事件。
- 文案草稿入口。

### 市场详情页

必须显示：

- 市场标题。
- 来源平台。
- 价格。
- 流动性。
- 成交量。
- 截止时间。
- 事件类型。
- 我方概率。
- 市场隐含概率。
- 差值。
- 行动建议。
- 取消条件。
- Polymarket 跳转链接，必须带联盟参数。

### 复盘页

必须显示：

- 结果。
- 盈亏。
- 判断是否正确。
- 价格是否正确。
- 仓位是否正确。
- 错误类型。
- 下一次调整。

## 文案原则

- 预测文案可以有观点，但必须保留条件。
- 每条推荐都要能被一句话解释。
- 不用夸张标题诱导下注。
- 可以写“最值得观察”，不要把观察伪装成推荐。
- 可以写“高盈亏比”，但必须说明低命中率或高风险可能。
- 第一版网站必须包含文案系统入口。

## 技术原则

- 用户明确提出的建设原则必须追加到 `platform/rules/system_rules.md`。
- 后续新增页面、脚本、模型或策略输出前，必须对照 `platform/rules/system_rules.md`。
- 先用文件和 SQLite 跑通，再做复杂服务。
- 先做只读数据，再做写入。
- 先做纸上交易，再考虑真实交易。
- 先做静态/本地工具，再考虑部署。
- 所有脚本必须可以单独运行。
- 输出必须保存到本地文件，方便复盘。
- 外链生成必须走统一函数或统一字段，避免漏掉联盟参数。
