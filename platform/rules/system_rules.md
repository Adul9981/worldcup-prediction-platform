# 系统建设规则库

这个文件记录用户明确提出、后续建设必须遵守的规则。每次新增功能、优化页面、调整模型或写脚本前，都要先对照本规则库。

## R001 预测市场选题优先

平台核心必须以预测市场中现存的可交易选题为中心，而不是以世界杯资讯、赛程新闻或泛分析为中心。

赛程、球队、小组、新闻和文案都是辅助层，必须服务于发现、解释和复盘预测市场交易机会。

## R002 选题类型必须分类

每个预测市场选题必须识别结构类型，至少包括：

- `binary_single_market`: 单个二元市场，例如 Yes / No。
- `multi_market_mutually_exclusive_group`: 多个二元市场组成的互斥组，例如同一冠军事件下多个队伍只能有一个最终胜出。
- `multi_choice_single_market`: 单市场多结果选项，预留给未来数据源。
- `non_exclusive_related_markets`: 不互斥但高度相关的选题，例如同队出线、夺冠、球员奖项等。
- `independent_binary`: 与当前主题无明显互斥或强相关的独立二元市场。
- `special_event_binary`: 特殊事件二元市场。
- `unknown_structure`: 暂无法判断，必须人工复核。

## R003 互斥关系必须进入风控

同一事件下互斥市场不能只看单个选题赔率。必须合并看：

- 同一 `event_slug` 下的总价格。
- 是否只有一个结果能成立。
- 是否存在 `Other` bucket。
- 是否存在 placeholder/team code。
- 是否 closed/resolved/excluded。
- 是否存在负风险或组合套利空间。

当前数据若缺少 Polymarket `neg_risk` 字段，必须标记为 `neg_risk_unknown`，不能假装已确认。

## R004 可以给下注方向，但必须有来源标记

当模型能够给出下注方向时，必须输出：

- `selection_direction`: YES / NO / WAIT / AVOID。
- `direction_confidence`: low / medium / high。
- `direction_source`: model / scoring_model / staking_model / manual / unavailable。
- `direction_thesis`: 一句话下注或等待理由。
- `cancel_condition`: 取消条件。

下注方向不是下单指令。用户可见层只输出方向和分析结果，不输出仓位建议、下注金额或单位数。

## R005 当前阶段默认只读

系统可以显示下注方向和分析结果，但不得连接钱包、不得下单、不得读取私钥，不得在用户可见层输出仓位建议。

## R006 网站第一屏要以市场机会为主线

世界杯赛程、阶段、小组、球队信息可以展示，但必须作为市场机会的解释层。页面主要入口必须回到：

- 当前可交易选题。
- 选题结构分类。
- 交易机会评分。
- 下注方向。
- 分析理由。
- 互斥/相关风险。
- Polymarket 跳转链接。

## R007 后续明确要求必须入库

如果用户后续明确提出建设原则、页面原则、模型原则、数据原则或风险原则，必须加入本规则库，并在后续优化时对照检查。

## R008 优化前后必须做规则对照

每次新增页面、脚本、模型或策略输出时，必须检查：

- 是否违反预测市场选题优先。
- 是否输出或保留选题结构。
- 是否说明互斥/相关风险。
- 是否区分买 YES、买 NO、等待、回避。
- 是否避免输出仓位建议。
- 如展示价格，是否同步展示回报率。
- 是否使用联盟链接。
- 是否仍然只读。

## R013 不输出仓位建议

用户明确要求：不要输出仓位建议，只给下注方向建议和分析结果。

因此用户可见页面、CSV、Markdown 摘要和文案中，不得展示：

- 建议仓位。
- 下注单位。
- 下注金额。
- 今日额度。
- 加仓/减仓数量。

内部风控模型可以继续作为方向判断的辅助信号，但输出层必须只显示：

- 方向：买 YES / 买 NO / 等待 / 回避。
- 置信度。
- 分析理由。
- 风险与取消条件。
- 选题结构和互斥关系。

## R014 价格必须配回报率

用户明确要求：如果页面展示价格，也必须展示对应回报率。

用户可见层的回报率用于理解价格空间，不代表下注金额、仓位大小或下单指令。

当前 YES 价格的默认回报率口径为潜在净回报率：

- `回报率 = (1 / YES价格 - 1) * 100%`

- 当前数据是否只是赛前/小组赛/淘汰赛/决赛阶段。

进度纵览不能替代预测市场机会主线，只能作为交易机会解释层。

## R015 当前建设优先级

用户明确排序：以下事项优先级较高，必须先推进。其中数据自动刷新优先级最高：

- 数据自动刷新。
- 新增/消失选题检测。
- 更完整的机会评分。
- 世界杯赛程和阶段数据。
- 每日机会工作台。

以下事项优先级较低，在高优先级功能稳定前延后：

- 价格变化榜。
- 交易量增长榜。
- 复盘系统。
- 文案系统。
- 订阅付费系统。
- 多平台接入。

## R016 低风险/高风险事件必须先过规则检查

所有用户可见的“低风险事件”和“高风险事件”展示前，必须同时满足：

- 来自预测事件库，而不是页面临时硬编码。
- 有明确策略说明 `strategy_summary`。
- 有明确结构说明 `structure_cn` 和互斥标记。
- 已检查是否违反 R001-R015。
- 不输出仓位、下注金额、下注单位。
- 如展示价格，必须展示回报率。
- Polymarket 链接必须使用邀请码链接。

低风险事件不能包含：

- `selection_direction = AVOID`。
- `status = excluded / closed / resolved`。
- 价格缺失且无明确策略。
- 互斥结构未说明。
- 只因为交易量高就被列入。

高风险事件不能等同于“推荐下注”。高风险事件必须标注为高回报观察、弱队相关、复杂互斥或待确认事件，并展示触发条件和取消条件。

任何没有通过规则检查的事件，只能进入普通观察或回避，不得出现在低风险/高风险重点模块。

## R017 今日下注方向必须优先今日比赛

“今日下注方向”模块必须优先展示今日比赛相关预测事件。

禁止把非今日比赛的已上架市场，仅因为 `selection_direction = YES`、交易量高或回报率高，排在今日下注方向前面。

今日下注方向的展示顺序必须是：

1. 今日比赛人工推荐策略。
2. 今日比赛已上架预测市场。
3. 今日比赛待监控事件，例如赛果、总进球、让球、双方进球、牌角球。
4. 单独分区展示“当前已上架机会”，不得混同为今日下注方向。

如果今日比赛相关市场尚未上架，必须明确显示“待监控”，不能用冠军盘或其他跨阶段市场补位。

## R018 高风险不等于极低概率

高风险事件不能简单等同于低价格、超高回报率或几乎不可能发生的事件。

以下事件不得仅凭高回报率进入高风险重点模块：

- 明显不可能或极低概率的冠军盘。
- 弱队夺冠这类缺少现实路径的事件。
- 只有价格极低但没有明确策略触发条件的事件。

高风险事件必须同时具备：

- 有现实交易路径。
- 有明确策略解释。
- 有明确取消条件。
- 不是 `AVOID`。
- 不是单纯靠赔率极低/回报率极高入选。

当前 V1 暂不纳入红牌、天气、裁判等临场因素。高风险优先包括：

- 次级强队挑战强队或强队互打的比赛，例如日本、挪威、摩洛哥、哥伦比亚、克罗地亚等对强队。
- 回报较高但路径合理的平局、受让、双方进球、总进球、小组路径事件。
- 市场容易因强队/明星队叙事被误判的对手方向或替代玩法。

## R019 低风险必须有可解释的稳健性

低风险事件不能仅凭模型买 YES 或流动性高入选。

低风险事件必须同时具备：

- 与今日比赛、明确强弱关系、明确赛程动机或高确定性结构相关。
- 有清晰策略理由。
- 有可解释的取消条件。
- 市场状态正常。
- 不属于长期冠军盘中低概率幻想事件。

冠军盘原则上不得直接归类为低风险，除非后续用户明确确认对应策略规则。

低风险 V1 允许包含以下阶段到达类事件：

- 明显强队进入淘汰赛。
- 明显强队进入32强。
- 明显强队进入16强。
- 明显强队进入八强。
- 明显强队进入半决赛。

这类事件必须标注为阶段到达策略，不能和冠军盘混为一类。

## R020 重点清单必须人工准入，默认空置

用户明确要求：低风险、高风险、每日关注的事件筛选宁可空着，不要乱加乱写。

因此以下用户可见重点清单必须采用人工准入或明确规则准入，不得由模型方向、球队强弱、价格、回报率、交易量、流动性、模板事件自动填充：

- 每日关注。
- 低风险事件。
- 高风险事件。
- 可选预测策略。

默认状态：

- 所有新生成的预测事件默认 `focus_bucket = 未入选`。
- 所有新生成的预测事件默认 `focus_admission_note = 未经过人工准入`。
- 页面重点清单只展示 `focus_bucket` 明确等于对应清单的事件。
- 如果没有通过准入的事件，页面必须显示空状态，不得使用冠军盘、阶段到达模板、强弱模板、次级强队模板或高回报模板补位。

准入前必须满足：

- 事件来自预测事件库或人工推荐库。
- 有明确事件边界、策略理由、触发条件、取消条件。
- 已确认不是仅凭价格、交易量、模型 YES、强队标签或高回报率入选。
- 今日关注必须与今日比赛、今日已上架市场或人工当日策略直接相关。
- 低风险必须说明低风险来自何处，而不是用“强队”替代逻辑。
- 高风险必须说明风险来源和可交易路径，而不是用“赔率高”替代逻辑。

后续填充这些清单时，必须先补规则或人工准入记录，再写入事件库；没有准入记录时保持空白。

用户明确要求：“今日下注方向页面”内容太多、太乱。

今日比赛列表必须能区分：

- 已结束。
- 今日未开始 / 进行中 / 待确认。
- 后续比赛。

如果比赛已经结束，今日页应标注“已结束”或不展示；不得把已结束比赛当作今日可操作策略。

## R028 前台信息架构采用三页主线

用户已同意前台页面规划：网站应做成克制的比赛日决策板，而不是复杂数据平台。

前台主页面只保留：

- 进度纵览：展示全部比赛状态，明确已结束、今日、后续和时间待确认。
- 预测事件库：展示事件总览和完整预测事件表，只做底层仓库。

页面职责：

- 今日页是首页和主入口。
- 今日推荐只来自用户后台手动录入。
- 今日比赛只显示尚未结束的北京时间今日比赛。
- 进度纵览负责赛程状态，不展示市场结构、方向分布、阶段统计等零散指标。
- 预测事件库负责完整事件查询，不展示推荐、低风险、高风险或可选策略卡片。

弱化或隐藏：

- 工作台只保留为简短入口页，不能承载复杂指标。
- 全部市场只作为原始数据/调试页，不进入主导航。
- 任何方向分布、结构统计、机会分类、交易量榜，除非后续明确证明对决策有用，否则默认删除。

## R029 前台减少解释性提示，增加克制的世界杯视觉

用户明确要求：网站展示不能过于朴素，可以增加足球或世界杯元素；同时删除类似“后台手动维护的重点策略，优先于模型自动清单展示”“只读版本”等提示备注文字。

前台页面规则：

- 不展示“只读版本”这类状态说明。
- 不展示后台维护、模型自动清单、下单提醒、回报率口径等解释性备注。
- 不在主要页面 footer 堆提示文案。
- 空状态文案必须短，不写长说明。
- 可以使用克制的足球/世界杯视觉元素，例如球场色、草坪纹理、比赛卡片、圆形足球线条等。
- 不做营销页，不做大段解释，不做花哨装饰；视觉元素必须服务比赛日阅读和快速识别。

后台页面规则：

- 后台只保留表单和当前置顶列表。
- 不展示“后台说明”等解释区。
- 不做格式校验提示。

## R030 今日推荐后台必须使用线上共享数据源

用户确认要做真正线上后台后，后台不得再只依赖浏览器 `localStorage` 作为发布数据源。

线上后台规则：

- 后台保存的今日推荐必须写入线上共享存储。
- 前台今日推荐必须优先读取线上 API，而不是当前浏览器缓存。
- 管理后台需要独立入口和管理口令保护。
- 用户换浏览器、换设备或其他人打开前台时，应看到同一份已发布今日推荐。
- 如果线上存储未配置，后台必须明确保存失败，不能假装发布成功。
- 发布包 JSON 只能作为降级兜底，不作为真正线上后台的主要数据源。
